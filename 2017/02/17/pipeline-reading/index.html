startup这个脚本里面主要是一堆的import，比如import了numpy, casa….,另外运行了EVLA_functions.py和lib_EVLApipeutils.py。这两个里面都定义和编写了一些pipeline运行过程中可能会用到的函数。

python里面的try..except语句是用来做异常处理的。try后面跟操作，但是异常发生，则执行except.这里出现try的动机可以理解为防止出现异常的编程模式。

EVLA_pipe_import.py是用来导入数据的。

EVLA_pipe_hanning.py是用来对数据进行hanning平滑的。可选操作。如果有窄带RFI的话会比较重要。这个脚本主要是执行casa的hanningsmooth命令。

EVLA_pipe_msinfo.py是用来收集数据的一些基本信息的，并画出一些统计图样。
这个脚本中主要用到的casa命令有listobs,plotms,plotants,flagcmd。

EVLA_pipe_flagall.py用来执行一些初始的flagging操作：
里面用到的flag模式有，summary,’antennas not on source’, shadow, zeros.
这里的这个online flags是什么(?)

EVLA_pipe_flagall.py每个spw的最后5%的channel丢掉，每个基带的最后10个channel丢掉。

EVLA_pipe_flagant.py用来flag掉不理想的天线的数据。

EVLA_pipe_calprep.py做一些校准之前的准备工作：把model数据的信息给主校准源。这里用到的命令是setjy

EVLA_pipe_priorcals.py用来做预校准。（因为总是校准-flag-校准…这么来的）这一步骤相当于我原先写的pipeline的“事先准备”部分
包括：
gain curves校准；
opacities校准；
天线位置校正；
requantizer gains (?)

用到的命令为’gc’的gencal. (L band的大气影响很小，所以不需要’opac’模式的gencal了)

EVLA_testBPdcals.py用来先看下目前得到的结果是啥样子——通过查看bandpass校准源和delay校准源。这里用到了gaincal, bandpass, plotcal.是用目前的还很raw的数据来做phase校准然后做bandpass校准，并且画出两个calibrator分别的amp和phase的solution的图。

注意这一步骤取消掉了，attila的笔记说是这一步骤对于有RFI的L-band数据效果非常糟糕。

EVLA_pipe_checkflag.py在这里用来rflag一下所有的校准源用以去除可能的RFI。这里被attila改成了两个iterations。
这里面的流程如下：

  1. 手动flag，应该会出来一个界面或者至少让输入flag的区间然后flag掉
  2. clip。这里设置了阈值为40 Jy
  3. rflag。阈值选取的sigma分别为4和3
  4. extend flagging

这里的EVLA_pipe_semiFinalBPdcals1.py用来做delay和bandpass校准。
这里包括：

  1. 重新选取reference antenna。（这一步骤应该是自动进行的，需要手工操作。动机是，经过前面那么一堆flag的折腾，说不定这个参考天线应该换一换了。）
  2. 对phase校准源进行G类型p模式的gaincal，并plotcal，横轴是频率纵轴是delay
  3. 对bandpass校准源进行G类型p模式的gaincal，scan被combine。然后plotcal，横轴是时间，纵轴是phase
  4. 对bandpass校准源执行B类型的bandpass操作。scan被combine。然后把bpsolution用plotcal画出来。横轴是频率纵轴是amp，以及横轴是频率纵轴是phase
  5. 把结果applycal到所有的校准源数据
  6. 把bandpass校准源的横轴频率纵轴amp的结果图plotms出来。

这里的EVLA_pipe_checkflag_semiFinal.py用来对calibrator做rflag。被attila修正为两个iterations.

  1. 先把校准源执行clip的flagging操作，阈值分别设置为40Jy何20Jy
  2. 对校准源rflag
  3. 对校准源extend flagging

再做semiFinalBPdcals.py。这个程序跟上面那个*BPcals1.py差别不大。先找到参考天线，然后做一系列的校准并画图。这里提到了说进阶这来个rflag，但是我在这个脚本里面没看到再次进行flag的程序段。

EVLA_pipe_solint.py是用来得到scan需要预先知道的solint的。注意solint的意思是solution interval. (?)

  1. 先split，把calibrator数据拿出来
  2. 然后分析(?)

EVLA_pipe_testgains.py的目的是DO TEST GAIN CALIBRATIONS TO SEE IF MORE FLAGGING IS NEEDED AND TO ESTABLISH SHORT AND LONG SOLINTS

  1. 重新确定参考天线
  2. 经过一些算法判断solint
  3. 画出gain calibrator的solution

EVLA_pipe_fluxgains.py用来做一个gain table，里面有gain和opacity的校准。这是为了最后的amp.cal服务，并最终为了flux density的bootstraping服务。(?)

  1. setjy
  2. G-type p-mode gaincal应用到所有校准源.
  3. G-type ap-mode gaincal应用到所有校准源。

啥叫bootstrapping?统计中对应的名词应该是“自举法”。简单说就是对于一个样本进行又放回的抽样，多次后形成一系列的样本，对于这些样本的分析可以得到原始样本统计量的分布。这里的考虑是，如果只有一个样本的话，可以得到相应的统计量，但是无法得到统计量的分布。

这里的EVLA_pipe_fluxboot.py是用来对所有的校准源的做谱流量自举分析并进行power law拟合，并把拟合结果加入到model列中 (?)

  1. fluxscale
  2. plotms

EVLA_pipe_finalcals.py用来制作最终的校准table

  1. 重新找一下参考天线
  2. G-type p-mode gaincal应用到delay calibrator
  3. K-type p-mode gaincal应用到delay calibrator。至此delay校准结束。
  4. G-type p-mode gaincal应用到bandpass calibrator，solint=soilnt_1
  5. B-type solint=inf的bandpass应用到bandpass calibrator
  6. 这一步，Derive an average phase solution for the bandpass calibrator to apply to all data to make QA plots easier to interpret，对于G-type p-mode的gaincal应用到bandpass calibrator，solint=inf
  7. unflag操作，对于上一步刚进行的gaincal。因为进行这些gaincal的时候，很不好的solution的数据可能会被flag掉。
  8. applycal于所有的calibrator
  9. split不同的calibrator
  10. 对于每一个calibrator执行setjy操作
  11. 对calibrator的数据进行G-type p-mode的gaincal得到phaseshortgaincal.g
  12. 对calibrator的数据进行G-type ap-mode的gaincal，solint=gain_solint_2得到finalampgaincal.g
  13. 对calibrator的数据进行G-type p-mode的gaincal得到finalphasegaincal.g
  14. plotcal一堆图。

EVLA_pipe_applycals.py把得到的solution应用到数据中并检查calibrated数据如何

  1. applycal

EVLA_pipe_targetflag.py是用来对包括target的所有数据做一次rflag的。

  1. clip flagging于所有的数据。对于两个calibrator设置的阈值是不同的，一个是40Jy一个是10Jy
  2. rflag应用于所有的数据
  3. extend仅应用于target数据
  4. plotms
  5. summary模式flagdata

EVLA_pipe_filecollect.py是用来整理得到的这些table和plots的，以便于下一步weblog的生成。

EVLA_pipe_weblog.py用来写入网络日志。各种write.
